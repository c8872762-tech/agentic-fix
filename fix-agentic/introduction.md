# Home Fix Agent — Project Introduction

## 1. Motivation

Every homeowner has been there. A light bulb burns out in the kitchen. You unscrew it, stare at the faded markings on the glass, and realize you have no idea what to buy. Is it E26 or E12? A19 or BR30? 60 watts or 60-watt equivalent? Dimmable? 2700K or 5000K? You drive to the hardware store, stand in an aisle of 200 options, and either guess wrong or give up and buy whatever looks close enough. The same problem repeats for batteries (CR2032 vs. CR2025), air filters (is it 20×25×1 or 20×25×4?), outlet covers, faucet cartridges, and dozens of other mundane household items.

The core issue is a knowledge gap. The information needed to buy the right replacement part — base type, wattage, dimensions, thread size, material — is physically present on the broken item or its surroundings, but most people cannot read it, interpret it, or translate it into a search query that returns the correct product. This is a problem that AI is uniquely positioned to solve: the input is visual, the reasoning is domain-specific but well-structured, and the output is a concrete, actionable product recommendation.

The Home Fix Agent is built on a simple premise: **a photo of the problem should be enough to find the solution**. A homeowner takes a picture of a burned-out bulb, a dead battery in a remote control, or a cracked outlet cover. The system analyzes the photo, extracts the technical specifications of the item that needs replacing, searches for matching products online, ranks them by relevance and value, and presents the top options for the user to review. If the user likes what they see, the system places the order — but only after the user explicitly confirms. No autonomous purchasing. No black-box decisions. The human stays in the loop at every step that involves spending money.

This project is motivated by three observations. First, multimodal LLMs have reached the point where they can reliably read text from photos, identify physical objects, and reason about their properties — exactly the capabilities needed to turn a photo of a broken bulb into a structured product specification. Second, product search APIs (Google Shopping, Amazon Product Advertising API) make it possible to programmatically find and compare products without scraping. Third, the agentic pattern — decomposing a complex task into specialized, composable steps with clear contracts between them — provides a natural architecture for a system that must perceive, reason, search, rank, and act in sequence, with human checkpoints along the way.

The target users are homeowners who lack hardware expertise, renters who need to replace items quickly and correctly, and property managers who handle maintenance across multiple units. The system is designed for common, low-complexity items: bulbs, batteries, filters, covers, knobs, and fasteners. It explicitly does not handle complex renovations, provide safety advice, or make autonomous purchasing decisions.


## 2. Agentic Strategy

The system is designed as a **sequential multi-agent pipeline** where each agent has a single, well-defined responsibility, strict input/output contracts, and no ability to bypass the human confirmation gate. The architecture follows the principle that no single agent should both recommend a product and place the order. This separation of concerns is not just good software design — it is a safety requirement. When real money is involved, the system must be transparent about what it knows, what it inferred, and what it is uncertain about, and it must stop and ask before acting.

The pipeline consists of five agents:

**Agent A: Vision Analyst.** This agent receives the raw photo and an optional text description from the user. It calls a multimodal LLM (GPT-4o) with a carefully constrained prompt that instructs it to describe only what is visible — the item category (bulb, battery, filter, hardware), the problem type (broken, burned out, cracked, missing), any readable text (brand names, model numbers, wattage markings), and the physical characteristics (shape, color, base type). The prompt explicitly forbids the agent from recommending products or guessing text it cannot read. The output is a structured JSON record with a confidence score. If the photo is blurry or ambiguous, the confidence drops and the system can ask the user for a better photo.

**Agent B: Spec Extractor.** This agent takes the Vision Analyst's output and the original photo, and determines the full replacement product specification. For a light bulb, that means base type (E26, GU10), wattage equivalent, color temperature, shape (A19, BR30), technology (LED, CFL), and whether it is dimmable. The agent assigns a confidence score to each field. When a critical field has confidence below 0.7, the agent generates a targeted clarification question — for example, "Is this bulb on a dimmer switch?" — rather than guessing. This clarification loop runs up to three rounds before the system proceeds with its best-effort specs. The agent also generates a search query string optimized for retailer APIs.

**Agent C: Product Searcher.** This agent takes the structured spec and queries external product APIs. In production, it calls SerpAPI's Google Shopping endpoint. For offline development and testing, it falls back to a curated mock dataset with realistic products across all supported categories. The searcher normalizes the raw API response into a consistent schema: title, price, rating, review count, availability, URL, and SKU.

**Agent D: Product Ranker.** This agent scores and ranks the search results using a weighted formula: 40% specification match (how many search terms appear in the product title), 20% customer rating, 20% price (lower is better), 10% availability, and 10% review count. The weights are config-driven and can be tuned without code changes. Each product receives a match score and a human-readable recommendation reason ("Strong spec match. Highly rated. In stock."). The ranker is deterministic — the same inputs always produce the same ranking.

**Agent E: Order Manager.** This agent handles the final confirmation and order placement. It creates a pending order record, displays the full order summary (product name, quantity, unit price, total, shipping), and waits for the user to explicitly type "yes" or click "confirm." Any other input — including an empty response — is treated as a cancellation. Only after receiving explicit confirmation does the agent place the order (via retailer API in production, or a mock placement that generates a tracking ID in the MVP). The order record is stored regardless of outcome, so cancelled orders are auditable too.

A key architectural decision is that the agents are implemented as **plain Python modules with strict JSON contracts**, not as autonomous agents in a multi-agent framework. Each agent is a function that takes structured input and returns structured output. The pipeline runner calls them in sequence. This design was chosen deliberately over more complex agent orchestration frameworks for three reasons: it is easier to debug (each step's input and output is inspectable), it is cheaper (no agent-to-agent communication overhead), and it supports deterministic replay (given the same inputs and model versions, the pipeline produces the same outputs). The "agentic" quality comes from the decomposition of the task into specialized reasoning steps with LLM calls, not from autonomous agent behavior.

The human-in-the-loop design is enforced at two levels. First, the clarification loop in the Spec Extractor gives the user a chance to correct misidentified specs before the system searches for products. Second, the confirmation gate in the Order Manager is a hard architectural boundary — the state machine that governs the ordering flow requires an explicit user confirmation to transition from `CONFIRMATION_PENDING` to `ORDER_PLACED`. There is no code path that bypasses this gate.


## 3. Test Approach

Testing an agentic system that depends on LLM calls and external APIs requires a layered strategy. The system cannot be tested end-to-end in a fully automated way — vision model outputs are non-deterministic, and product search results change daily. Instead, the test approach isolates what can be tested deterministically from what requires human evaluation, and provides mock infrastructure so that the deterministic layers can run without API keys or network access.

**Unit tests** cover the foundational components that have no LLM or API dependencies. The test suite includes 18 automated tests organized into five groups:

- *Schema validation* (4 tests): Verifies that all Pydantic data models — Session, IssueAnalysis, ProductSpec, ProductResult, OrderRecord, PipelineResult — instantiate correctly with default values, serialize to JSON, and deserialize without error. This catches schema regressions early.

- *Photo intake* (4 tests): Validates the input boundary. The system must reject files that are not JPEG, PNG, or HEIC; reject files over 20 MB; raise a clear error for missing files; and correctly copy valid photos to the session storage directory. These tests use temporary files and do not require real photos.

- *Ranking engine* (4 tests): Verifies that the weighted scoring formula is deterministic (same inputs produce same outputs), that products with higher spec match rank higher, that the normalization function handles edge cases (all-equal values, inverted normalization for price), and that rank numbers are assigned sequentially. These tests use synthetic ProductResult objects and do not call any API.

- *Order manager* (4 tests): Confirms that the confirmation gate works correctly. Creating an order sets status to `PENDING`. Confirming an order sets status to `PLACED`, generates a mock retailer order ID, and records a timestamp. Cancelling sets status to `CANCELLED`. Quantity multiplication is correct (3 units at $5.00 = $15.00 total).

- *Mock search* (2 tests): Verifies that the mock fixture file loads correctly and that the mock search function returns properly typed ProductResult objects for a given query. This ensures that the entire pipeline can run end-to-end without a SerpAPI key.

**Integration tests** are designed to run the full pipeline against fixture data. A sample photo is fed through the Vision Analyst and Spec Extractor (which require an LLM API key), then through the Product Searcher (using mock data), Ranker, and Order Manager. The test verifies that the pipeline produces a complete PipelineResult with analysis, specs, ranked products, and a storable session. This test requires an OpenAI API key and is run manually rather than in CI.

**LLM contract tests** verify the behavioral constraints of the LLM agents. The Vision Analyst must return valid JSON matching the IssueAnalysis schema; it must not recommend products (only describe what it sees); and it must set confidence low when the image is unclear. The Spec Extractor must return valid JSON matching the ProductSpec schema; it must generate clarification questions when confidence on a critical field is below 0.7; and its search_query field must be non-empty. These tests are semi-automated: the JSON schema validation is checked programmatically, but the behavioral constraints (e.g., "does not recommend products") are verified by human review of the LLM output on a set of test photos.

**The mock infrastructure** is a critical part of the test strategy. The system includes a curated JSON fixture file with realistic product data for four categories: bulbs (5 products including Philips, Amazon Basics, Cree, GE, and Sylvania), batteries (3 products: Energizer, Duracell, Amazon Basics), filters (3 products: Filtrete, Nordic Pure, Honeywell), and hardware (3 products: Leviton, Enerlites, Lutron wall plates). When no SerpAPI key is configured, the Product Searcher automatically falls back to this mock data, matching the query to the most relevant category. This means the entire pipeline — from photo upload through product ranking and order placement — can be exercised without any external API calls except the LLM vision call.

**Manual evaluation** uses a test matrix of five photo scenarios: a burned-out A19 bulb with visible text (should extract E26 base and 60W), a CR2032 battery in a remote control (should extract the correct size), an HVAC filter with dimensions printed on the frame (should match the printed size), a blurry photo of a ceiling light (should report low confidence and ask for a better photo), and a photo of a faucet handle (should identify the item category without crashing, even though faucets are an edge case). Each scenario has explicit pass criteria. This matrix is run after each significant change to the LLM prompts or the pipeline logic.

**Session replay** provides a final layer of testability. Every pipeline run stores all intermediate artifacts — the raw photo, the Vision Analyst output, the extracted specs, the search results, the ranked products, and the order record — as JSON in a session directory. The `replay` command loads a past session and displays its full output. This makes it possible to inspect exactly what happened at each step when a recommendation was wrong, identify which agent made the error (was the spec extraction wrong? was the search query bad? did the ranker score incorrectly?), and re-run the pipeline on the same inputs after a fix to verify the improvement. This replay capability is not just a debugging tool — it is the foundation for the offline evaluation loop that will drive continuous improvement of the prompts, the ranking weights, and the category-specific spec extraction logic.
